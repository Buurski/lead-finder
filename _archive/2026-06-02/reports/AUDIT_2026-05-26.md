# Lead-System Audit — 2026-05-26

Dybt eftersyn af hele lead-systemet (kode + scheduled tasks + send pipeline + scrape). Sorteret efter impact.

## 🔴 KRITISKE — fix denne uge

### 1. chains.ts apostrof-bug rammer produktion
**Filer:** `src/lib/chains.ts:4-10`, `src/app/api/email/bulk-send/route.ts:43`

`isChain("Bone's Restaurant")` returnerer `false` fordi `\bbones\b` ikke matcher "bone's" (apostrof breaker word-boundary). Production bulk-send-route bruger isChain direkte → en Bone's-lead med email + score ≥ 50 vil blive emailet til.

**Fix:** I `CHAIN_EXACT` ændre `"bones"` → `"bone's"`. Også verificer at `"papa john's"`, `"domino's"`, `"sticks'n'sushi"` matcher korrekt (regex tester `\bX\b` med X = literal string — apostrof er ikke word-char).

**Test:** `isChain("Bone's Aalborg")` skal returnere true efter fix.

### 2. send.mjs har hardcodede Gmail credentials
**Fil:** `.send_queue/send.mjs:16-17`

```js
const GMAIL_USER = "buur.aigro@gmail.com";
const GMAIL_PASS = "xahr qovv dnus jche";
```

Hvis denne fil bliver checked ind i git, comm'er via screen share, eller deler du Workflows-mappen → credentials lækket. .env.local er gitignored, send.mjs er IKKE.

**Fix:** Lad scriptet læse fra `.env.local` (samme loader-pattern som .fresh_scrape.mjs).

### 3. send.mjs opdaterer ikke sheet → follow-ups virker ikke
**Filer:** `.send_queue/send.mjs:101`, `src/app/api/email/send-followups/route.ts:18`

send.mjs skriver kun til `state.json`. Sheet's kolonne O (emailSentAt) er aldrig sat for de 39 sendte emails. send-followups bruger `lead.emailSentAt` → ingen follow-ups vil nogensinde fyre for noget vi sender via .send_queue.

**Fix:** Skriv `.send_queue/.sheet_updater.mjs` der kører som background process og syncer state.json → sheet col O + R hver minut. Eller integrer i send.mjs direkte efter hvert sendte email.

## 🟠 HØJ — fix indenfor 2 uger

### 4. scoreLead inflaterer FB-only leads
**Fil:** `src/lib/apify.ts:166-170`

```js
if (!place.website) score += 30;  // no-website bonus
```

Tjekker kun om `website` er null. En lead med `website: "facebook.com/foo"` har en website-string → får IKKE bonussen. Men en lead med `website: ""` (rent FB-only uden indtastet URL) får bonussen, selvom FB-only er svært at konvertere.

**Fix:** Tilføj eksplicit penalty for FB-only:
```js
if (!place.website) score += 30;
else if (/facebook\.com|instagram\.com/i.test(place.website)) score -= 10;
```

### 5. .fresh_scrape.mjs taber 49% af branche-tags
**Fil:** `.send_queue/.fresh_scrape.mjs:155-165`

Når Google Places returnerer "Hovedentreprenør", "Tjenester", "Byggemarked" osv. for craft-søgninger, kasseres viden om HVILKEN søgning der hentede dem. Morning-task'ens filter matcher kun specifikke trades — 49% af nattens leads kunne ikke retaggerm.

**Fix:** I `runScraper` loop, sæt `place.searchedBranch = branch` (fra BRANCH_PRESETS-iteration) før dedupe. I appendLeads, brug `searchedBranch` som branch (capitalize første bogstav) i stedet for `categoryName`. Drop retag-pass helt.

### 6. CITIES mangler hele København-området
**Fil:** `src/lib/apify.ts:31-50`

Ingen København, Frederiksberg, Lyngby, Hørsholm, Charlottenlund, Birkerød, Glostrup, Hvidovre, Albertslund, Roskilde, Helsingør, Hillerød. Massivt untapped marked (~2M mennesker).

**Fix:** Tilføj `cph` region preset:
```js
cph: ["København", "Frederiksberg", "Lyngby", "Hørsholm", "Charlottenlund", "Birkerød", "Glostrup", "Hvidovre", "Albertslund", "Roskilde", "Helsingør", "Hillerød", "Allerød", "Farum", "Værløse", "Ballerup", "Herlev", "Rødovre", "Vanløse", "Brøndby", "Greve", "Køge", "Taastrup", "Tårnby"],
```

### 7. Inkonsistent score-tærskel mellem bulk-send og morning-task
**Filer:** `bulk-send/route.ts:48`, morning SKILL.md step 4

Bulk-send kræver score ≥ 50, morning-task kræver score ≥ 40. Samme lead kan godkendes af morning men afvises af bulk-send. Lucas bruger ikke bulk-send længere → cleanup vil reducere forvirring.

**Fix:** Slet bulk-send route eller align tærskel til 40.

### 8. verify-all klassificerer FB-only som "dead"
**Fil:** `src/app/api/verify-all/route.ts:49-52`

Returnerer `tier: "dead"` for FB-only sites. Morning-task SKILL.md har separate `fb_only` klassifikation. Resultat: sheet col L viser "dead" → morning-task overrider via egen klassifikation → forvirring.

**Fix:** Tilføj `"fb_only"` til WebsiteQualityTier-typen i sheets.ts, return den fra verify-all.

## 🟡 MEDIUM — backlog

### 9. queue.json vokser unbounded
**Fil:** `.send_queue/queue.json`

Hver send-cyklus appender til samme fil. Nu 65 entries (alle sendt). Næste batch bliver 65+N. Om en måned: 1000+. Hurtigere I/O hvis vi roterer.

**Fix:** Efter send.mjs exit, archive queue.json til `archive/queue_{date}.json` og start frisk.

### 10. Ingen sync af messenger_state til sheet
**Fil:** `lead-messenger-morning/SKILL.md` step 5e

Hvis laptop crash eller .send_queue tabes, mistes Messenger-historik. Sheet kolonne R sættes til "messenger-sent" — OK, men hvilken FB-handle og timestamp? Gemmes kun lokalt.

**Fix:** Opret nyt sheet-tab "Messenger" med kolonner: rowIndex, fbHandle, name, sentAt, replied, repliedAt. Skriv hver send dertil parallelt med messenger_state.json.

### 11. bulk-find-emails MAX_PER_RUN = 100, men 366 leads mangler email
**Fil:** `src/app/api/email/bulk-find-emails/route.ts:11`

Skal køres 4 gange manuelt for at dække alle. Lucas kører den slet ikke længere (morning-task håndterer email-hunting per-batch).

**Fix:** Slet eller adapter til scheduled task der kører ugentligt.

### 12. send.mjs poll-loop spilder strøm
**Fil:** `.send_queue/send.mjs:78-122`

Når queue er tom, poll'er den hver 2 min i 5 cykler før exit. Hvis Lucas tilføjer ny batch midt i poll-fasen → den fanges. Men hvis ingen ny → 10 min med Node-process der bare venter.

**Fix:** Skift til event-driven: brug `fs.watch(QUEUE_FILE)` til at vække scriptet ved ændring i stedet for polling. Eller bare exit straks når tom og lad næste batch genstarte processen.

### 13. AGENTS.md + CLAUDE.md er stale
